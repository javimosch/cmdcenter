- **Error**: Container restart loop with "no such file or directory"
- **Fix**: Switched to debian:stable-slim (glibc compatible)
- **Impact**: Required base image change, increased image size (~74MB vs ~7MB for Alpine)
- **Severity**: Medium (resolved)

- **Severity**: Medium (needs code fix)

## Comparison: Build vs Binary Deployment

### Build-Based Deployment (compose.yml)
- **Base Image**: Multi-stage (golang:1.23-alpine → alpine:latest)
- **Build Time**: ~80 seconds
- **Image Size**: ~20MB (optimized multi-stage)
- **Advantages**:
  - Self-contained (no external binary dependency)
  - Smaller final image
  - Reproducible builds
  - No architecture mismatch issues
- **Disadvantages**:
  - Requires build step on deployment
  - Requires Go toolchain in build environment
  - Longer deployment time

### Binary-Based Deployment (compose.binary.yml)
- **Base Image**: debian:stable-slim
- **Build Time**: 0 seconds (no build)
- **Image Size**: ~74MB (Debian base)
- **Advantages**:
  - Fast deployment (no build step)
  - Simple compose file
  - Uses pre-tested binary
  - Easy to update binary independently
- **Disadvantages**:
  - Larger image size
  - External binary dependency
  - Architecture compatibility concerns
  - Must ensure binary matches base image libc

## Performance Metrics

### Container Startup Time
- **Build-based**: ~5 seconds (after build)
- **Binary-based**: ~3 seconds (no build)
- **Winner**: Binary-based (faster deployment)

### Image Size
- **Build-based**: ~20MB
- **Binary-based**: ~74MB
- **Winner**: Build-based (smaller image)

### Deployment Complexity
- **Build-based**: Medium (requires build step)
- **Binary-based**: Low (no build step)
- **Winner**: Binary-based (simpler)

### Reproducibility
- **Build-based**: High (source → image)
- **Binary-based**: Medium (binary dependency)
- **Winner**: Build-based (more reproducible)

## Recommendations

### For Development
- **Use**: Build-based deployment (compose.yml)
- **Reason**: Reproducible, smaller images, self-contained

### For Production
- **Use**: Binary-based deployment (compose.binary.yml) with static binary
- **Reason**: Faster deployments, simpler compose files
- **Requirement**: Build static binary with `CGO_ENABLED=0`

### For Multi-Architecture
- **Use**: Build-based deployment with docker buildx
- **Reason**: Can build for multiple architectures in one step

### Hybrid Approach
- **Development**: Build-based for testing
- **Production**: Binary-based for speed
- **CI/CD**: Build binary in pipeline, deploy with compose.binary.yml

## Hotify Integration

### Current Approach (Incorrect)
Hotify was responsible for Docker configuration - this is wrong.

### Correct Approach
Hotify should only manage which compose file to use, similar to Coolify:
- Hotify knows about multiple compose files (compose.yml, compose.binary.yml, compose.dev.yml)
- Hotify can switch between compose files
- Hotify can start/stop containers using specified compose file
- Docker configuration remains cmdcenter's responsibility
